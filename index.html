Question 01
Compare File Processing System with Database Management System 10 Marks

What is a File Processing System?

A File Processing System (FPS) stores data in separate, isolated files managed by different application programs. Each application has its own files and there is no central control or management of data.

Example: Student records stored in one Excel file, fee records in another, attendance in another — all separate, no connection between them.

What is a DBMS?

A Database Management System (DBMS) is software that provides a centralised, organised, and controlled way to store, retrieve, and manage data. It acts as an interface between users and the database.

Example: MySQL, Oracle, PostgreSQL — all data stored centrally, accessible by multiple users simultaneously.

Problems with File Processing System

Data RedundancySame data stored in multiple files — student name stored in attendance file, fees file, and marks file separately
Data InconsistencyUpdating one file does not automatically update others — address changed in one file but not others
Data IsolationData scattered across files — hard to access related data together
No Concurrent AccessMultiple users cannot access or update the same file at the same time
No SecurityNo access control — anyone who can open the file can see everything
No Backup/RecoveryManual backup — no automatic crash recovery mechanism
No Data IndependenceApplication code depends on file structure — changing the file breaks the application

Question 02
Explain Data Independence — Logical and Physical 10 Marks

What is Data Independence?

Data Independence is the ability to modify the schema at one level of the database architecture without affecting the schema at the next higher level. It ensures users do not need to know how data is physically stored or logically structured in the backend.
💡 Analogy
Think of a TV remote. You press a button (external view) without knowing how the signal is sent internally. If the manufacturer changes the circuit inside (internal schema), you still use the same remote the same way. That separation is data independence.

1. Logical Data Independence

The ability to modify the Conceptual Schema (logical structure) without affecting the External Schema (user views) or application programs.

What changes are covered:

Change at Conceptual LevelEffect on External Level
Adding a new tableNo effect on existing user views
Deleting an existing tableOnly affected views are updated
Adding a new column to a tableNo effect on other user views
Modifying relationships between tablesNo effect on user views

Example: In a university database, if the DBA adds a new table called Hostel or adds a new column CGPA to the Student table — a professor's view showing only salary and attendance remains completely unaffected.

Note: Logical Data Independence is harder to achieve because changing the logical structure often has a deeper impact.

2. Physical Data Independence

The ability to modify the Internal Schema (physical storage) without affecting the Conceptual Schema (logical structure).

What changes are covered:

Change at Internal LevelEffect on Conceptual Level
Changing file structure (linked to sequential)No effect on tables or relationships
Adding or removing indexesNo effect on logical structure
Changing storage pathsNo effect on conceptual schema
Compressing or reorganising data on diskNo effect on logical structure

Example: If the DBA changes the database storage from linked access to sequential access — the Student table, Course table and all relationships remain completely unaffected.

Note: Physical Data Independence is easier to achieve because storage changes are well isolated.

Question 03
Describe the Overall Architecture of DBMS with Diagram 10 Marks

What is DBMS Architecture?

DBMS architecture defines how users interact with the database and how data is stored, accessed, and managed. It can be centralised (data at one location) or decentralised (data distributed across multiple locations).

Part A — 3-Tier Client-Server Architecture

Tier 1 — Single Tier Architecture

The client, server, and database all reside on the same machine. The user directly accesses and manages data without any network. Used by database designers and programmers for development and testing.
┌────────────────────────────────────┐ │ Single Machine │ │ ┌────────┐ ┌────────┐ ┌──────┐ │ │ │ User │──│ Server │──│ DB │ │ │ └────────┘ └────────┘ └──────┘ │ └────────────────────────────────────┘ Used by: Developers, Programmers, Testers

Tier 2 — Two Tier Architecture

A client application sits between the user and the database server. The user sends requests through the application which communicates directly with the database server. Multiple users can access simultaneously but performance degrades under heavy load.
┌──────────────────────┐ ┌──────────────────┐ │ CLIENT SIDE │ │ SERVER SIDE │ │ ┌──────────────┐ │ │ ┌────────────┐ │ │ │ User + App │◄───┼─request─┼──► Database │ │ │ └──────────────┘ │◄─result─┼──┤ │ │ └──────────────────────┘ └──────────────────┘

Tier 3 — Three Tier Architecture

Three distinct layers: client, application logic (middle tier), and database server. The client never directly touches the database — all requests go through the middle tier which contains business logic.
┌─────────────┐ ┌──────────────────────┐ ┌──────────────┐ │ CLIENT TIER │ │ APPLICATION TIER │ │ DATA TIER │ │ │ │ (Middle Tier / │ │ │ │ User + UI │◄──►│ Business Logic) │◄──►│ Database │ │ Browser/App │ │ App Server │ │ Server │ └─────────────┘ └──────────────────────┘ └──────────────┘ Layer 1 Layer 2 Layer 3

Part B — 3-Schema Architecture (ANSI/SPARC)

Data independence is achieved through three levels of logical abstraction. Users should not need to know how data is physically stored — they just see what they need to see.

Layer 1 — External Schema (View Level)

This is what individual users see. Different users get different views of the same database. A student sees only their own marks; a professor sees salary and class records. Multiple external schemas can exist for different user roles.

Layer 2 — Conceptual Schema (Logical Level)

The complete logical blueprint of the entire database — all tables, relationships, constraints, and ER models. Managed by the DBA. Any change here does not affect the external schema — this is Logical Data Independence.

Layer 3 — Internal Schema (Physical Level)

Defines how data is physically stored — file structures, indexes, storage paths, access methods. Changes here do not affect the conceptual schema — this is Physical Data Independence.

Question 05
Define with Example — Different Types of Entities in ER Diagram 10 Marks

What is an Entity?

An entity is any real-world object or concept that has a distinct existence and about which data can be stored in a database. Entities are represented as rectangles in ER diagrams.
💡 Analogy
Entities are like nouns in English. STUDENT, EMPLOYEE, BOOK — anything you can say "I want to store data ABOUT this" is an entity. Just like every noun can be a subject — every entity can be a table.

1. Strong Entity

A strong entity has its own Primary Key and can exist independently — it does not depend on any other entity for identification.
PropertyDetail
Has own Primary Key?Yes — uniquely identifies itself
Depends on another entity?No — fully independent
ER SymbolSingle Rectangle (one border)
ExampleSTUDENT (StudentID, Name, Age) — StudentID uniquely identifies it

Strong Entity Diagram

2. Weak Entity

A weak entity does NOT have its own Primary Key. It depends on a Strong Entity (its Owner) for identification. It uses a Partial Key (Discriminator) combined with the owner's PK.
💡 Analogy
A DEPENDENT (family member) in a company database has no Employee ID of their own. They are identified only by their name + their employee's ID. Without the employee, the dependent record makes no sense. That's a Weak Entity.
PropertyDetail
Has own Primary Key?No — uses Partial Key + Owner's PK
Depends on another entity?Yes — depends on Strong Entity (Owner)
ER SymbolDouble Rectangle (two borders)
Relationship with OwnerIdentifying Relationship — Double Diamond
ExampleDEPENDENT (Name, Relationship) depends on EMPLOYEE (EmpID)

Weak Entity Diagram

3. Associative Entity (Composite Entity)

Created from a M:N relationship between two entities when the relationship itself has attributes. It resolves Many-to-Many into two 1:N relationships.
PropertyDetail
Created fromM:N Relationship that has its own attributes
ER SymbolRectangle inside Diamond
ExampleENROLLMENT (StudentID, CourseID, Grade, Date) from STUDENT M:N COURSE

4. Recursive Entity (Unary Relationship)

A recursive entity participates in a relationship with itself. The same entity type appears more than once in the relationship.
💡 Analogy
A manager is also an employee. So EMPLOYEE manages EMPLOYEE. Same entity, pointing back to itself.

Relational Algebra Operations with Examples 10 Marks

What is Relational Algebra?

Relational Algebra is a procedural query language that takes one or two relations as input and produces a new relation as output. It forms the theoretical foundation of SQL.

Sample Table — EMPLOYEE

EmpIDEmpNameDeptSalaryCity
101AliceIT50000Mumbai
102BobHR30000Delhi
103CharlieIT70000Mumbai
104DavidFinance40000Pune

1. Select Operator (σ)

Selects rows (tuples) that satisfy a given condition. It is a horizontal filter — filters rows, keeps all columns.

Syntax: σ condition (Relation)

Example: σ Dept='IT' (EMPLOYEE)

SQL Equivalent: SELECT * FROM EMPLOYEE WHERE Dept = 'IT'

EmpIDEmpNameDeptSalaryCity
101AliceIT50000Mumbai
103CharlieIT70000Mumbai

Multiple conditions: σ Dept='IT' AND Salary>60000 (EMPLOYEE) → Returns only Charlie

2. Project Operator (π)

Selects specific columns (attributes) from a relation. It is a vertical filter — filters columns, keeps all rows. Duplicate rows are automatically removed.

Syntax: π attribute_list (Relation)

Example: π EmpName, Dept (EMPLOYEE)

SQL Equivalent: SELECT EmpName, Dept FROM EMPLOYEE

EmpNameDept
AliceIT
BobHR
CharlieIT
DavidFinance

Combined: π EmpName, Salary (σ Dept='IT' (EMPLOYEE)) → First filter rows, then select columns

3. Union Operator (∪)

Combines tuples from two relations and removes duplicates. Both relations must be union-compatible (same number of attributes, same domains).

Syntax: Relation1 ∪ Relation2

Example: IT_EMP ∪ MUMBAI_EMP

If IT_EMP = {Alice, Charlie} and MUMBAI_EMP = {Alice, Charlie} → Result = {Alice, Charlie} (no duplicates)

SQL Equivalent: SELECT * FROM IT_EMP UNION SELECT * FROM MUMBAI_EMP

4. Rename Operator (ρ)

Renames a relation or its attributes. Useful when performing self-joins or when you need to refer to the same relation with two different names.

Syntax: ρ new_name (Relation) OR ρ new_name(attr1, attr2) (Relation)

Example 1 — Rename table: ρ EMP (EMPLOYEE) → EMPLOYEE is now referred to as EMP

Example 2 — Rename table and attributes: ρ EMP(ID, Name, Department, Sal, Location) (EMPLOYEE)

Example 3 — Self join using Rename (find employees in same city):

ρ E1 (EMPLOYEE) and ρ E2 (EMPLOYEE) σ E1.City = E2.City AND E1.EmpID ≠ E2.EmpID (E1 × E2)

SQL Equivalent: SELECT * FROM EMPLOYEE AS EMP

5. Set Difference (−)

Returns tuples that are in the first relation but NOT in the second relation. Both relations must be union-compatible.

Syntax: Relation1 − Relation2

Example: ALL_EMP − IT_EMP

If ALL_EMP = {Alice, Bob, Charlie, David} and IT_EMP = {Alice, Charlie} → Result = {Bob, David}

SQL Equivalent: SELECT * FROM ALL_EMP EXCEPT SELECT * FROM IT_EMP

6. Cartesian Product (×)

Combines every tuple of one relation with every tuple of another relation. If R has m tuples and S has n tuples, R × S has m×n tuples.

Syntax: Relation1 × Relation2

Example: EMPLOYEE × DEPARTMENT

If EMPLOYEE has 4 rows and DEPARTMENT has 3 rows → Cartesian Product = 4×3 = 12 rows

SQL Equivalent: SELECT * FROM EMPLOYEE, DEPARTMENT (no join condition)

What are Aggregate Functions?

Aggregate functions perform calculations on a set of rows and return a single value. They are used to summarise and analyse data in a table.

Sample Table — EMP

EmpIDEmpNameSalaryDept
101Alice50000IT
102Bob30000HR
103Charlie50000IT
104David70000HR
105EveNULLIT

1. SUM()

Definition: Returns the total sum of all values in a column. NULL values are ignored.

SELECT SUM(salary) FROM emp; -- Output: 200000 SELECT SUM(DISTINCT salary) FROM emp; -- Output: 150000 (50000 counted once, not twice)

2. AVG()

Definition: Returns the average value. Calculated as SUM/COUNT. NULL values are ignored.

SELECT AVG(salary) FROM emp; -- Output: 50000 (200000/4, NULL row ignored) SELECT AVG(DISTINCT salary) FROM emp; -- Output: 50000 (150000/3)

3. COUNT()

Definition: Returns the number of rows.

VariantBehaviourOutput
COUNT(*)Counts ALL rows including NULL5
COUNT(salary)Counts only non-NULL rows4 (Eve's NULL not counted)
COUNT(DISTINCT salary)Counts unique non-NULL values3 (30000, 50000, 70000)
SELECT COUNT(*) FROM emp; -- Output: 5 SELECT COUNT(salary) FROM emp; -- Output: 4 SELECT COUNT(DISTINCT salary) FROM emp; -- Output: 3

4. MAX()

Definition: Returns the highest value in a column. NULL values are ignored.

SELECT MAX(salary) FROM emp; -- Output: 70000 SELECT dept, MAX(salary) FROM emp GROUP BY dept;

5. MIN()

Definition: Returns the lowest value in a column. NULL is treated as "data not available" — not as zero — so it is ignored.

SELECT MIN(salary) FROM emp; -- Output: 30000 (NULL ignored, not treated as 0) SELECT dept, MIN(salary) FROM emp GROUP BY dept;

Question 09
Write SQL Queries for Employee Database 10 Marks

Schema

Employee (empname, street, city, date_of_joining) Works (empname, company_name, salary) Company (company_name, city) Manages (empname, manager_name)

Q1. Modify database so that 'John' now lives in 'Mumbai'

UPDATE employee SET city = 'Mumbai' WHERE empname = 'John';

Logic: UPDATE sets a new value. WHERE targets only John's row. Without WHERE, ALL rows would be updated.

Q2. Find all employees who joined in the month of October

SELECT empname, date_of_joining FROM employee WHERE MONTH(date_of_joining) = 10;

Logic: MONTH() extracts the month number from a date. October = 10. Similarly YEAR() and DAY() extract year and day.

Q3. Give all employees of 'ABC Corporation' a 10% raise

UPDATE works SET salary = salary * 1.10 WHERE company_name = 'ABC Corporation';

Logic: Multiplying by 1.10 adds exactly 10% to current salary. (salary × 1.10 = salary + 10% of salary)

Q4. Find employees who earn more than average salary of their company

SELECT w1.empname, w1.company_name, w1.salary FROM works w1 WHERE w1.salary > ( SELECT AVG(w2.salary) FROM works w2 WHERE w2.company_name = w1.company_name );

Logic: This uses a correlated subquery. w1 and w2 are aliases of the same Works table. The inner query calculates the average salary of ONLY w1's company. The outer query keeps employees whose salary exceeds that average. The subquery runs once for every row in the outer query.

Q5. List names of companies starting with letter "A"

SELECT company_name FROM company WHERE company_name LIKE 'A%';

Logic: LIKE is used for pattern matching. 'A%' means: starts with A, followed by anything.

PatternMeaningMatches
LIKE 'A%'Starts with AABC, Amazon, Accenture
LIKE '%A'Ends with ATATA, Infosysa
LIKE '%A%'Contains A anywhereTCS, Amazon, BAT
LIKE 'A_'Starts with A + exactly 1 more characterAB, AC

SQL Queries on Employee and Company Tables — Updates, Joins, Salaries, Cities 10 Marks

Schema

Employee (emp_id, emp_name, city, salary, dept) Company (company_id, company_name, city, industry)

UPDATES

-- Update employee city UPDATE employee SET city = 'Mumbai' WHERE emp_name = 'Alice'; -- Give 20% raise to IT department UPDATE employee SET salary = salary * 1.20 WHERE dept = 'IT'; -- Give flat 5000 bonus to all employees UPDATE employee SET salary = salary + 5000; -- Update company city UPDATE company SET city = 'Bangalore' WHERE company_name = 'Wipro';

JOINS

-- Employee name + company name (same city JOIN) SELECT e.emp_name, c.company_name FROM employee e JOIN company c ON e.city = c.city; -- Employee name + company + industry (same city) SELECT e.emp_name, c.company_name, c.industry FROM employee e JOIN company c ON e.city = c.city; -- Left Join — show all employees even if no company in their city SELECT e.emp_name, e.city, c.company_name FROM employee e LEFT JOIN company c ON e.city = c.city;

CONDITIONS ON SALARY

-- Employees earning more than 40000 SELECT emp_name, salary FROM employee WHERE salary > 40000; -- Employees earning between 30000 and 60000 SELECT emp_name, salary FROM employee WHERE salary BETWEEN 30000 AND 60000; -- Employees earning more than overall average SELECT emp_name, salary FROM employee WHERE salary > (SELECT AVG(salary) FROM employee); -- Average salary per department SELECT dept, AVG(salary) AS avg_salary FROM employee GROUP BY dept; -- Highest and lowest salary SELECT MAX(salary) AS highest, MIN(salary) AS lowest FROM employee;

CONDITIONS ON CITIES

-- Employees in Mumbai SELECT emp_name, city FROM employee WHERE city = 'Mumbai'; -- Employees NOT in Delhi SELECT emp_name, city FROM employee WHERE city != 'Delhi'; -- Employees in Mumbai or Pune SELECT emp_name, city FROM employee WHERE city IN ('Mumbai', 'Pune'); -- Employees and companies in SAME city SELECT e.emp_name, e.city, c.company_name FROM employee e JOIN company c ON e.city = c.city; -- Employees in DIFFERENT city than any company SELECT e.emp_name, e.city AS emp_city, c.company_name, c.city AS company_city FROM employee e JOIN company c ON e.dept = c.industry WHERE e.city != c.city;

Why is Normalisation Needed? Explain 1NF, 2NF, 3NF, BCNF with Examples 10 Marks

What is Normalisation?

Normalisation is the process of organising a database to reduce redundancy and improve data integrity by dividing large tables into smaller, well-structured tables.

Why is Normalisation Needed? — Anomalies

AnomalyMeaningExample

Question 13
Describe ACID Properties with Examples and State Transition Diagram 10 Marks

What is a Transaction?

A transaction is a logical unit of work performed on a database. It is an action or series of operations performed by a single user or application that reads or modifies database content. A transaction must either be fully committed or fully rolled back — there is no partial execution.

ACID Properties Overview

PropertyFull FormCore Idea
AAtomicityAll or nothing
CConsistencyData remains correct before and after
IIsolationConcurrent transactions do not interfere
DDurabilityCommitted data persists permanently

A — Atomicity

A transaction must be fully completed and committed, or if it fails midway, all work must be rolled back. There is no partial execution.
💡 Example — IRCTC Ticket Booking
When booking a train ticket, if payment of ₹1000 is debited but ticket booking fails due to a network error — Atomicity ensures the ₹1000 is automatically rolled back. Either the full transaction completes (payment + ticket) or nothing happens at all.

C — Consistency

A transaction must bring the database from one valid consistent state to another. The integrity and correctness of data must be maintained before and after the transaction.
💡 Example — Bank Transfer
When ₹1000 is transferred: your account is debited ₹1000 AND the recipient account is credited ₹1000. Total money before = Total money after. Data integrity is maintained — this is Consistency.

I — Isolation

Multiple transactions can execute concurrently without interfering with each other. One transaction's intermediate state is not visible to other transactions until it is fully committed.
💡 Example — Last Seat Train Booking
Only 1 seat is left on a Mumbai-Delhi train. Rahul and Priya try to book the same seat at the same time. Due to Isolation: Rahul's transaction executes independently, Priya cannot see or interfere until Rahul commits. Once Rahul's booking commits, seat is marked unavailable and Priya's booking fails. Same data, concurrent access, no corruption.

D — Durability

Once a transaction is committed, it is permanently saved to the database and disk. Even if the system crashes immediately after, the data will persist.
💡 Example — UPI Payment
You make a UPI payment and it shows successful. Seconds later the bank server crashes. When the server comes back online, your account statement still shows the transaction as successful — because it was already committed to disk.

DurabilityCommitted data never lostDisk Storage / Write-Ahead Logs

Question 14
Explain Lock-Based (2PL) Concurrency Control Method with Example 10 Marks

What is Concurrency Control?

Concurrency control ensures that multiple transactions execute simultaneously without causing data inconsistency. Without it — Lost Updates, Dirty Reads, and Unrepeatable Reads can corrupt your database.

What is Two-Phase Locking (2PL)?

Two-Phase Locking is a concurrency control protocol that ensures serializability by dividing transaction execution into exactly two strict phases.
💡 Analogy — Train Seat Booking
Two people try to book the last seat simultaneously. Without a lock, both see the seat as available and both book it — 2 people, 1 seat, chaos. A lock ensures only one person can hold the seat at a time. Once one person commits, the seat is unavailable to others. That's 2PL.
PhaseNameRule
Phase 1Growing PhaseTransaction can ONLY acquire locks. Cannot release any lock. Locks are requested as needed.
Phase 2Shrinking PhaseTransaction can ONLY release locks. Cannot acquire any new lock. Once first lock is released — no more locking ever.

Types of Locks

Lock TypeSymbolAllowsUsed When
Shared LockS-LockMultiple transactions can READ simultaneouslySELECT / Read
Exclusive LockX-LockOnly ONE transaction can READ and WRITEINSERT / UPDATE / DELETE

Lock Compatibility Matrix

Requested ↓ / Held →S-Lock HeldX-Lock Held
S-Lock RequestCompatible — both can readNot Compatible — Wait
X-Lock RequestNot Compatible — WaitNot Compatible — Wait

Example — 2PL in Action

Two transactions T1 (Bank Transfer) and T2 (Balance Check) on Account A (balance = ₹1000):

TimeT1 (Transfer)T2 (Balance Check)Lock Status
t1X-Lock(A) — acquired—T1 holds X-Lock on A
t2Read A = ₹1000S-Lock(A) → WAITT2 blocked — X incompatible with S
t3Write A = ₹500Still waitingT1 still in Growing Phase
t4Unlock(A) — releaseS-Lock(A) — ACQUIREDT1 enters Shrinking Phase
t5—Read A = ₹500T2 sees committed correct value

2PL Diagram — Growing and Shrinking Phases

Number of Locks | Lock Point (max locks held) | ↑ | ____↑____ | / \ | / \ | / \ | / \ |─────────────────────────────────→ Time Growing | Shrinking (Acquire) | (Release only)

Variants of 2PL

VariantRuleAdvantage
Basic 2PLGrow then shrinkEnsures serializability
Strict 2PLHold all X-locks until commit/abortPrevents dirty reads — most commonly used
Rigorous 2PLHold ALL locks (S and X) until commitStrictest — easiest recovery
Conservative 2PLAcquire all locks before transaction startsNo deadlocks but impractical

Problem with 2PL — Deadlock

2PL can cause deadlocks when two transactions wait for each other's locks forever.

T1 holds Lock(A) and waits for Lock(B) T2 holds Lock(B) and waits for Lock(A) → Neither can proceed — DEADLOCK Solved using: Wait-Die or Wound-Wait protocols

Timestamp-Based Protocol

Each transaction is assigned a unique timestamp TS(T) when it starts. Older transactions (smaller TS) have higher priority. The protocol ensures transactions execute in timestamp order — as if they ran serially in that order.
💡 Analogy — Bank Token System
At a bank, Token 5 (issued earlier = older) has priority over Token 20 (issued later = younger). If Token 20 tries to jump ahead, it is sent back and given a new token. That's exactly how the Timestamp Protocol works — older transactions have priority.

Two Timestamps Per Data Item

TimestampSymbolMeaning
Read TimestampR-TS(X)Timestamp of the LAST transaction that successfully READ X
Write TimestampW-TS(X)Timestamp of the LAST transaction that successfully WROTE X

Rules

When Transaction T wants to READ data item X:

ConditionActionReason
TS(T) < W-TS(X)ROLLBACK TT is too old — trying to read a newer version
TS(T) ≥ W-TS(X)Allow READ — update R-TS(X)T is reading correct version

When Transaction T wants to WRITE data item X:

ConditionActionReason
TS(T) < R-TS(X)ROLLBACK TNewer transaction already read X
TS(T) < W-TS(X)ROLLBACK TNewer transaction already wrote X
OtherwiseAllow WRITE — update W-TS(X)Safe to write

Example

TransactionTimestampOperationResult

T2TS = 200Read(X)Allowed — TS(200) ≥ W-TS(100). R-TS(X) = 200
T3TS = 50Write(X)ROLLBACK — TS(50) < R-TS(200). T3 is too old.
T4TS = 300Write(X)Allowed — W-TS(X) = 300

Advantages vs Disadvantages

AdvantagesDisadvantages

Question 16
Explain Conflict and View Serializability with Example 10 Marks

What is Serializability?

A schedule (sequence of operations from multiple concurrent transactions) is serializable if its outcome is equivalent to some serial schedule where transactions execute one after another without overlap.
💡 Analogy
Imagine two chefs cooking in the same kitchen at the same time. The result must taste the same as if Chef A cooked alone first, then Chef B cooked alone. If the concurrent result matches the serial result — the schedule is serializable.

Part A — Conflict Serializability

Two operations conflict if: they belong to different transactions AND they operate on the same data item AND at least one of them is a WRITE.
Operation PairConflict?Reason
Read(X) by T1, Read(X) by T2NoBoth are reads — no conflict
Read(X) by T1, Write(X) by T2YesOne is write on same item
Write(X) by T1, Read(X) by T2YesOne is write on same item
Write(X) by T1, Write(X) by T2YesBoth write same item

Precedence Graph Method

1. Draw one node for each transaction.
2. Draw a directed edge Ti → Tj if Ti has a conflicting operation that appears BEFORE Tj's operation.
3. If graph has NO cycle → Conflict Serializable.
4. If graph has a cycle → NOT Conflict Serializable.

Example Schedule:

TimeT1T2
t1Read(A)—
t2—Read(A)
t3Write(A)—
t4Read(B)—
t5—Write(A)
t6—Write(B)

Identifying Conflicts:

On BT1 Read(B) at t4, T2 Write(B) at t6RW ConflictT1 → T2

Precedence Graph:

No cycle → Schedule is CONFLICT SERIALIZABLE. Equivalent serial schedule: T1, T2.

Part B — View Serializability

Two schedules are view equivalent if: (1) Initial reads are the same, (2) Read-from relationships are the same, (3) Final writes are the same. View Serializability is a superset of Conflict Serializability.
ConditionMeaning
Initial ReadIf Ti reads initial value of X in S, it must read initial value in serial schedule too
Updated ReadIf Ti reads value written by Tj in S, same must hold in serial schedule
Final WriteThe transaction that does the final write of X must be same in both schedules

NOT every View Serializable is Conflict SerializableView Serial is a larger set

View Serializability — harder to testNP-complete in general

Write a Short Note on Log-Based Recovery Mechanism 5 Marks

What is Log-Based Recovery?

Log-based recovery records every modification made by transactions in a log file BEFORE applying changes to the actual database. This is called Write-Ahead Logging (WAL). In case of a crash, the log is used to restore the database to a consistent state.
💡 Analogy
Before a nurse gives any medicine, it is written in the hospital log book: Patient, Medicine, Old dosage, New dosage. If something goes wrong, the doctor checks the log and reverses the medicine. The database log works exactly the same way.

Log Record Structure

Format: <T, X, V1, V2>

FieldSymbolMeaning
Transaction IDTWhich transaction made the change
Data ItemXWhich data item was modified
Old ValueV1Value BEFORE change — used for UNDO
New ValueV2Value AFTER change — used for REDO

Types of Log Records

RecordMeaning
<Ti, start>Transaction Ti has started
<Ti, X, V1, V2>Ti changed X from V1 to V2
<Ti, commit>Ti successfully committed ✅
<Ti, abort>Ti was rolled back ❌
<checkpoint>All committed transactions flushed to disk up to this point

UNDO vs REDO

OperationWhen UsedWhat It DoesUses
UNDOTransaction NOT committed at crashRestores old values — rolls back changesOld Value V1
REDOTransaction committed but not on diskReapplies new values to databaseNew Value V2

Example

Log EntryMeaning
<T1, start>T1 begins
<T1, A, 1000, 1500>T1 changes A from 1000 to 1500
<T1, commit>T1 committed ✅ → will be REDONE
<T2, start>T2 begins
<T2, B, 500, 300>T2 changes B from 500 to 300
[CRASH]System crashes — T2 has no commit → will be UNDONE

Recovery Action:

  • REDO T1 — T1 has commit record. Re-apply A = 1500 to disk.
  • UNDO T2 — T2 has no commit record. Restore B = 500 (old value from log).

Recovery Process After Crash

StepAction
1. Scan LogRead log from last checkpoint forward
2. REDO ListTransactions WITH <commit> record → add to REDO list
3. UNDO ListTransactions WITHOUT <commit> record → add to UNDO list
4. REDO PhaseRe-apply all committed transactions in forward order
5. UNDO PhaseRoll back all uncommitted transactions in reverse order

Diagram

──────────────────────────────────────────────────────→ Timeline [checkpoint] ── [T1 start] ── [T1 commit] ── [T2 start] ── [CRASH 💥] ↑ ↑ T1 has commit T2 no commit → REDO T1 → UNDO T2 A = 1500 ✅ B = 500 ✅

Checkpointing

A checkpoint guarantees all committed transactions are written to disk at that point. During recovery, only transactions after the last checkpoint need to be processed — this significantly reduces recovery time.

Key Rule to remember: Has <commit> → REDO | No <commit> → UNDO | Write-Ahead Logging = Log FIRST, then update database.
